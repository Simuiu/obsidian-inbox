# Obsidian Inbox 中国市场内容营销与自动化方案

版本：v0.1 草案  
日期：2026-05-07  
目标：供人工审阅和反馈，确认后再进入物料制作与自动化执行。

## 1. 方案结论

Obsidian Inbox 的中国市场传播不应该被包装成“又一个 AI 总结工具”，而应该定位为：

> 给 Obsidian 装一个本地知识入库员：把视频、文章、本地资料变成可追溯、可索引、可复用的知识库。

第一阶段目标不是大规模涨粉，而是找到 100-300 个高质量早期用户：

- Obsidian / Logseq / Notion 知识管理用户
- Codex / AI Agent 重度用户
- 课程学习、投研、咨询、产品、技术学习人群
- 开源工具爱好者
- 有本地视频、会议录音、长文章整理需求的人

传播策略：

- 用 B 站和公众号建立“可信度”和完整解释。
- 用小红书、视频号、抖音做“痛点前后对比”和轻量触达。
- 用知乎、掘金、V2EX、即刻做“方法论 + 技术可信度 + 早期反馈”。
- 用自动化系统完成 70%-80% 的内容生产、排期、复用、复盘；发布动作按平台能力分为“自动发布”“半自动发布”“人工发布”三类。

## 2. 核心定位

### 2.1 一句话

中文：

> Obsidian Inbox：把视频、文章、本地资料自动整理成可追溯、可索引、可复用的 Obsidian 知识库。

英文：

> Obsidian Inbox turns videos, articles, and local media into traceable, indexed, reusable Obsidian notes.

### 2.2 差异化

| 常见 AI 总结工具 | Obsidian Inbox |
|---|---|
| 生成一次性摘要 | 生成可长期维护的知识库条目 |
| 结果散落在对话里 | 写入 Obsidian Vault |
| 原文和转写容易丢 | `_resources/` 保留可追溯材料 |
| 只有一个摘要层级 | 普通总结 / 深度总结 |
| 很难后续查找 | 资料库索引 / 主题索引 / 用途索引 |
| 常把自动化讲成全自动 | 明确区分自动、半自动、人工确认边界 |

### 2.3 不宣传什么

为了降低合规和信任风险，不宣传：

- “全平台一键下载”
- “自动绕过登录/验证码/付费墙”
- “无人值守全自动运营”
- “保证视频号 URL 直接下载”
- “替代人工判断”

对外统一表达：

> 这是一个 local-first 的 Codex + Obsidian 入库工作流。能自动化的自动化，涉及平台权限、发布审核、账号安全的部分保留人工确认。

## 3. 用户分层

### 3.1 第一优先级：知识管理用户

典型画像：

- 已经使用 Obsidian、Logseq、Notion、Heptabase
- 有大量收藏、课程、文章、视频资料
- 痛点是“收藏很多，但很少沉淀”

核心话术：

> 不是给你更多摘要，而是帮你把资料放进一个以后还能找回来的结构里。

### 3.2 第二优先级：AI 工具和 Agent 用户

典型画像：

- 会用 Codex、ChatGPT、Claude、Cursor
- 愿意尝试 skill、agent、MCP、本地脚本
- 对“local-first”“可验证”“开源”敏感

核心话术：

> 这是一个可复用的 Codex Skill，不是一次性提示词。

### 3.3 第三优先级：课程学习和专业研究用户

典型画像：

- 看长视频、课程、会议、直播回放
- 需要把内容变成学习笔记、方法论、SOP

核心话术：

> 一个本地视频进去，一篇带转写追溯的 Obsidian 深度笔记出来。

## 4. 平台策略

## 4.1 B 站

目标：

- 做主阵地之一，建立演示可信度。
- 承接“Obsidian + AI + 开源工具”用户。

内容类型：

- 5-8 分钟完整演示
- 15-20 分钟教程
- 2-3 分钟局部功能短视频
- 动态同步 GitHub 更新

首批选题：

1. 《我做了一个 Codex Skill：把视频和文章自动整理进 Obsidian》
2. 《别再只让 AI 总结了：把资料真正入库》
3. 《一个本地视频，如何变成一篇可追溯 Obsidian 深度笔记》
4. 《Obsidian + Codex：搭一个自己的知识入库员》

自动化建议：

- 自动生成视频脚本、分镜、字幕、封面文案。
- 自动生成剪映/CapCut 可用的旁白稿和字幕 SRT。
- 自动导出 B 站标题、简介、标签、评论区置顶文案。
- 发布动作建议人工确认。B 站投稿后台更适合人工上传和检查封面、分区、标签、转载声明。

## 4.2 小红书

目标：

- 用“前后对比”破圈。
- 获取知识管理、AI 工具、效率工具用户。

内容类型：

- 6-9 页图文卡片
- 30-60 秒录屏短视频
- 工具清单式笔记

首批选题：

1. 《AI 总结完还是找不到？我把它接进 Obsidian 了》
2. 《视频课太多看不完？我做了个自动入库工具》
3. 《Obsidian 用户最需要的不是模板，是入库流程》
4. 《我的 AI 学习资料库长这样》

自动化建议：

- 自动生成小红书图文卡片文案。
- 自动生成每页卡片布局说明。
- 自动生成标题 A/B 测试候选、正文、话题标签。
- 自动生成封面图提示词或 SVG/PNG 资产。
- 自动生成发布清单，但发布动作建议人工确认。

注意：

- 小红书官方创作服务平台提供创作和管理后台，但公开稳定的“个人账号自动发笔记 API”并不适合直接假设可用。
- 不建议使用非官方 cookie 自动发帖作为主方案；可以作为“小龙虾浏览器自动化的实验能力”，但必须人工确认、低频执行，并准备账号风控预案。

## 4.3 微信公众号

目标：

- 承载完整方法论。
- 做可长期引用的中文介绍页。
- 承接视频号和朋友圈传播。

内容类型：

- 2500-4000 字长文
- 产品故事
- 工作流拆解
- 更新日志

首批选题：

1. 《AI 总结之后，知识应该去哪？》
2. 《我为什么做 Obsidian Inbox：从收藏夹到资料库》
3. 《普通总结和深度总结：AI 笔记工作流应该如何分层？》

自动化建议：

- 可通过微信公众号草稿箱/发布接口链路做“Markdown -> HTML -> 上传图片 -> 创建草稿”的自动化。
- 最终群发/发布建议人工确认，避免格式、封面、错别字、合规风险。
- 自动化优先级：先做“自动进入草稿箱”，不要直接自动群发。

参考边界：

- 微信公众号官方文档提供草稿箱、发布等接口体系，实际使用需要公众号权限、access token、图片素材上传等配置。

## 4.4 视频号

目标：

- 承接微信生态。
- 用短视频演示“一个视频变成一篇 Obsidian 笔记”。

内容类型：

- 30-90 秒录屏演示
- 公众号长文摘要版
- 产品更新短视频

自动化建议：

- 自动生成短视频脚本、口播稿、字幕、封面。
- 自动生成公众号关联文案。
- 发布动作建议人工确认。

## 4.5 知乎

目标：

- 占领搜索和理性讨论场景。
- 通过问题回答获取高质量用户。

内容类型：

- 回答
- 专栏文章
- 想法

问题方向：

- 如何用 AI 做知识管理？
- Obsidian 有哪些值得推荐的工作流？
- AI 总结工具真的有用吗？
- 如何把视频课程整理成笔记？

自动化建议：

- 自动监控关键词问题。
- 自动生成回答初稿。
- 自动生成“方法论版”和“项目介绍版”两个版本。
- 发布建议人工确认，尤其避免硬广。

## 4.6 掘金 / CSDN / 开源中国

目标：

- 获取开发者和开源用户。
- 解释架构、边界和实现细节。

内容类型：

- 技术文章
- 架构拆解
- 开源项目介绍

首批选题：

1. 《如何设计一个 local-first 的 Codex + Obsidian 入库工作流》
2. 《Obsidian Inbox 架构拆解：task package、compact.md、索引重建》
3. 《为什么这个工具不上传你的 Vault、cookies 和本地视频》

自动化建议：

- 自动生成 Markdown 长文。
- 自动生成 Mermaid/架构图。
- 自动生成摘要、标签、封面。
- 发布动作可半自动：生成平台适配稿，人工复制发布。

## 4.7 即刻 / V2EX / 少数派 Matrix

目标：

- 获取早期用户反馈。
- 建立真实开发者形象。

内容类型：

- Show project
- 开发日志
- 求反馈帖

自动化建议：

- 自动生成不同语气版本：
  - 即刻：轻量真实
  - V2EX：技术直接
  - 少数派：工作流完整
- 发布建议人工确认，评论区高优先级人工回复。

## 4.8 抖音

目标：

- 做更大众化的“前后对比”短视频。
- 不作为首发主阵地，作为复用 B 站/视频号素材的分发渠道。

自动化建议：

- 抖音开放平台有视频上传/创建能力，但需要申请 `video.create` 权限并完成用户授权。
- 在权限申请完成前，采用“自动生成视频 + 人工发布”。
- 权限完成后，可以考虑“自动上传为草稿/待发布”，仍保留人工审核。

## 5. 自动化分级

### 5.1 L0：人工

适用：

- 最终发布确认
- 评论区高价值回复
- 平台敏感内容判断
- 选题方向调整

### 5.2 L1：素材自动生成

自动生成：

- 标题
- 正文
- 标签
- 摘要
- 封面文案
- 图文卡片文案
- 视频脚本
- 分镜
- 口播稿
- SRT 字幕
- GitHub 更新摘要

这是第一阶段必须完成的自动化。

### 5.3 L2：跨平台改写和排期

自动完成：

- 从一个母稿生成 B 站、公众号、小红书、知乎、掘金、V2EX 版本。
- 生成发布时间建议。
- 生成发布 checklist。
- 生成 Notion/Obsidian/飞书排期表。

### 5.4 L3：半自动发布

适用：

- 微信公众号：自动进草稿箱，人工发布。
- 抖音：申请权限后自动上传视频，人工发布。
- 小红书/B站：先生成发布包，人工上传。

### 5.5 L4：自动复盘

自动采集或人工录入：

- 阅读量
- 播放量
- 点赞
- 收藏
- 评论
- GitHub stars
- GitHub issues
- clone / fork 趋势

自动输出：

- 周报
- 高表现内容复盘
- 下周选题建议
- 评论区 FAQ 候选

## 6. 小龙虾协同方案

这里假设“小龙虾”是你已有或计划使用的内容自动化/Agent 工具。若它的实际能力不同，可以在确认阶段调整。

### 6.1 小龙虾的角色

建议把小龙虾定位为：

> 内容运营执行层：负责定期生成、改写、排期、提醒、复盘；不直接越权发布敏感平台内容。

它不负责：

- 改核心代码
- 直接登录平台做高风险发布
- 自动回复争议评论
- 绕过平台限制

它负责：

- 读取内容素材库
- 生成多平台内容包
- 建立排期
- 生成待发布清单
- 生成复盘报告
- 整理用户反馈

### 6.2 与 Obsidian Inbox 的内容流水线

```text
GitHub README / docs / issue / release
        ↓
内容母稿生成器
        ↓
小龙虾平台适配器
        ↓
多平台发布包
        ↓
人工确认 / 半自动发布
        ↓
数据回收
        ↓
复盘报告
        ↓
下一轮选题
```

### 6.3 小龙虾任务列表

每日：

- 检查 GitHub 是否有新 commit、issue、star 变化。
- 生成 1 条即刻/朋友圈/动态短内容候选。
- 检查评论和反馈是否需要人工回复。

每周：

- 生成 2 条小红书图文选题。
- 生成 1 条 B 站短视频脚本。
- 生成 1 篇知乎回答初稿。
- 生成 1 篇技术社区短文。
- 输出一份周复盘。

每月：

- 汇总内容数据。
- 生成版本更新文章。
- 更新 FAQ。
- 生成下一月内容日历。

### 6.4 小龙虾输入输出规范

输入目录建议：

```text
marketing/
├── briefs/
├── source/
├── platform-rules/
├── calendar/
├── drafts/
├── publish-packets/
└── reports/
```

输出包建议：

```text
publish-packets/YYYY-MM-DD-topic/
├── brief.md
├── bilibili.md
├── xiaohongshu.md
├── wechat-official-account.md
├── zhihu.md
├── juejin.md
├── v2ex.md
├── captions.srt
├── cover-prompts.md
└── checklist.md
```

## 7. 自动化系统设计

### 7.1 内容母稿

每个主题先生成一份母稿：

```text
主题：

核心痛点：
核心观点：

CTA：
风险边界：
```

母稿确认后再自动改写为不同平台版本。

### 7.2 平台适配器

每个平台有自己的适配规则：

| 平台 | 标题 | 正文 | 图片/视频 | CTA | 发布方式 |
|---|---|---|---|---|---|
| B 站 | 信息明确 | 教程/演示 | 横屏视频 | GitHub + 评论置顶 | 人工发布 |
| 小红书 | 痛点强 | 卡片化 | 竖图/短视频 | 轻 CTA | 人工发布 |
| 公众号 | 方法论 | 长文 | 图文 | GitHub + 视频 | 草稿箱半自动 |
| 视频号 | 短句 | 字幕 | 竖屏视频 | 公众号链接 | 人工发布 |
| 知乎 | 问题导向 | 理性回答 | 少图 | 低硬广 | 人工发布 |
| 掘金 | 技术标题 | 架构拆解 | 图/代码 | GitHub | 人工发布 |
| V2EX | 直接 | 求反馈 | 少图 | GitHub | 人工发布 |
| 抖音 | 强前后对比 | 字幕化 | 竖屏视频 | 评论区 | 半自动待定 |

### 7.3 发布日历

建议首月节奏：

| 周期 | 周一 | 周三 | 周五 | 周末 |

## 10. 自动化建设优先级

### P0：马上做

- 建立 `marketing/` 目录结构。
- 建立内容母稿模板。
- 建立多平台改写模板。
- 建立发布包格式。
- 建立发布 checklist。
- 建立首月内容日历。

### P1：确认后做

- 自动读取 GitHub README、docs、release，生成内容母稿。
- 自动把母稿改写为各平台发布稿。
- 自动生成封面图提示词。
- 自动生成短视频脚本和字幕。
- 自动生成周报。

### P2：后续做

## 11. 平台自动化边界

## 13. 风险与控制

| 风险 | 控制方式 |
|---|---|
| 被理解为盗链/下载工具 | 强调只处理可访问内容和本地材料，不绕过限制 |
| 被理解为全自动运营工具 | 强调内容生成自动化，发布保留人工确认 |
| 小红书/B站账号风控 | 不默认使用非官方自动发布 |
| AI 生成内容过度营销 | 每个平台保留人工审稿 |
| 用户不会安装 | 做 B站完整演示 + FAQ |
| 开源项目看起来太技术 | 小红书/视频号讲前后对比，少讲代码 |

## 14. 需要你确认的问题

1. “小龙虾”的实际能力是什么？
   - 是浏览器自动化？
   - 是内容生成 Agent？
   - 是定时任务平台？
   - 是否能读取本地仓库和写文件？

2. 首发时你是否愿意露脸或出声？
   - 不露脸不出声：录屏 + 字幕 + AI 配音
   - 不露脸出声：录屏 + 口播
   - 露脸：产品可信度更高，但制作成本更高

3. 是否建立反馈群？
   - 不建群：只导 GitHub issue
   - 建群：公众号/小红书更容易转化，但管理成本增加

4. 首发案例用什么？
   - B站技术视频
   - 本地课程视频
   - 微信公众号文章
   - 不建议首发用交易类视频，容易偏离工具定位

5. 自动发布边界你能接受到哪一级？
   - A：只自动生成物料，全部人工发布
   - B：公众号进草稿箱，其他人工
   - C：部分平台浏览器辅助填写，人工点发布
   - D：有 API 的平台自动发布，无 API 的人工
