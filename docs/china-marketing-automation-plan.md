# Obsidian Inbox 外部引流与自动化推广协作计划

## 1. 当前目标与原则

`Obsidian Inbox` 的 GitHub 基础内容和本地入库流程已经具备。当前阶段的重点不是继续堆功能说明，而是通过外部渠道为 GitHub 项目引入真实访问、Star、Issue 和试用反馈，同时验证一套“Codex 生成内容 + 你审核确认 + API/MCP/CLI 尽量自动发布或进草稿 + 数据复盘”的推广流程。

对外统一定位：

> 把 Codex 变成你的本地知识入库员：链接进来，Obsidian 笔记、原始材料和可点击索引自动生成。

核心原则：

- 先验证方向，再扩大执行。
- 内容先由 Codex 生成和自评，再交给你最终确认。
- 能通过官方 API、MCP 接口或 CLI 工具完成的，优先让 Codex 自动完成到“草稿/待发布/可检查”状态。
- 平台不支持稳定自动化时，采用半自动发布：Codex 生成发布包、检查清单和复制内容，你只做最终确认和必要操作。
- 每个平台的内容都要指向 GitHub 项目，GitHub 是统一承接页。
- 不把项目宣传成“全平台一键下载”“绕过限制抓取”“无人值守自动运营”。
- 账号、token、cookie、授权文件等敏感信息不直接写进对话；需要授权时使用本地配置、官方登录或你明确指定的安全方式。
- 如果同类推广流程重复 3 次以上，再考虑沉淀成 Skill。

## 2. 营销方向判断

外部工具类项目更容易通过“清晰场景 + GitHub 承接 + 社区反馈 + 可复用工作流”获得早期关注，而不是一开始做泛流量投放。

`Obsidian Inbox` 的传播重点不应该是“AI 总结”，而应该是“内容入库工作流”。用户真正需要的不是一次性摘要，而是把视频、文章、本地资料变成以后还能找回、能追溯、能复用的 Obsidian 资料库。

参考对象与启发：

| 对象 | 观察 | 对 `Obsidian Inbox` 的启发 |
| --- | --- | --- |
| Obsidian 生态插件 | 用户重视本地文件、Markdown、可迁移和长期维护 | 宣传时要强调 Vault-native、Markdown、索引和资源归档 |
| Anthropic / Codex Skills | Skill 适合沉淀可重复执行的流程 | 不要只讲工具，要讲“Codex + Obsidian 入库工作流” |
| awesome-claude-code 类项目 | 高质量索引可以帮助用户快速理解 AI 工具生态 | GitHub README 要先讲清适用人群、场景和安装路径 |
| SuperClaude / Claude Code Framework 类项目 | 通过 README、教程、社区讨论建立可信度 | 需要用技术长文讲清 task package、compact.md、_resources 和索引重建 |
| 常见 AI 总结工具 | 多数结果停留在对话或单篇摘要 | 差异化要放在“可追溯、可索引、可复用”，不是“总结更快” |

本阶段渠道判断：

| 渠道 | 作用 | 先后判断 | 自动化策略 |
| --- | --- | --- | --- |
| GitHub | 统一承接页，承接 Star、Issue、安装和试用 | 先做好 | README、Topics、Release、Issue 模板可生成草稿，提交前人工确认 |
| 公众号 / 知乎 | 讲完整故事，沉淀长期内容 | 先做长文稿 | 优先尝试官方接口、MCP 或 CLI 进入草稿；不稳定时生成发布包，人工发布 |
| B 站 | 降低理解成本，展示真实演示过程 | 第二优先级 | 生成脚本、分镜、标题、简介、标签；如有可靠上传工具则进草稿/待发布，否则人工投稿 |
| 掘金 | 建立技术可信度 | 第三优先级 | 优先尝试 CLI/API/浏览器辅助进入草稿；不行则生成 Markdown 发布包 |
| V2EX / 即刻 | 获取开发者和早期用户真实反馈 | 第四优先级 | 默认人工确认发布，Codex 准备标题、正文、回复预案和风险提示 |
| 小红书 / 视频号 | 测试知识管理、AI 工具、效率工具人群兴趣 | 第五优先级 | 先生成图文卡片文案和短视频脚本；如能安全进入草稿则由 Codex 辅助 |

## 3. 我们的角色分工

| 角色 | 职责 |
| --- | --- |
| Codex 执行者 | 生成渠道内容包、标题、正文、视频脚本、发布说明、风险提示、发布记录模板 |
| Codex 评审者 | 站在营销专家、目标用户、平台用户视角，判断内容是否可信、是否像广告、是否能打动人、是否适合发布 |
| Codex 发布助手 | 查找并使用可用的官方 API、MCP 接口、CLI 工具或低风险浏览器辅助方式，尽量把内容推进到草稿、待发布或可检查状态 |
| 你 | 做最终方向确认、口吻确认、发布风险确认、是否进入下一阶段的决策 |

协作标准：

- Codex 交付给你的内容，必须已经过自评，不把粗糙初稿直接丢给你。
- 你反馈后，Codex 先修改当前阶段内容，不跳到下一阶段。
- 只有你确认“这一阶段可以”，才进入下一阶段。
- Codex 优先减少你的手工操作：能自动保存草稿就保存草稿，能生成可直接复制的发布包就生成发布包，能用 CLI/API 查询结果就自动查询。
- Codex 不直接索要或展示敏感凭证；需要账号授权时，先说明需要什么权限、用于哪个平台、会写入哪里、如何撤销。
- 最终发布、涉及账号风险的动作、争议评论回复，仍由你确认后执行。
- 目标是让你每轮需要提的意见越来越少。

## 4. 全流程协作计划

| 阶段 | 阶段目标 | Codex 负责 | Codex 自评 | 你负责 | 进入下一阶段条件 |
| --- | --- | --- | --- | --- | --- |
| 1. 统一口径 | 定清楚所有渠道共用的卖点和表达边界 | 输出统一定位、核心卖点、禁用表达、GitHub 引流口径 | 判断是否清楚、可信、不过度夸大 | 确认方向和表达边界 | 你确认口径可用于所有渠道 |
| 2. 发布能力确认 | 先确认哪些平台能自动进草稿，哪些只能半自动 | 检查可用 API、MCP、CLI、官方后台、授权要求和风险边界 | 判断是否稳定、是否安全、是否值得接入 | 提供必要账号授权或决定放弃自动化 | 每个平台形成“自动/半自动/人工”结论 |
| 3. 长文内容包 | 先做公众号/知乎长文，用来讲完整故事 | 输出长文大纲、正文、标题候选、摘要、发布备注，并尽量进入草稿 | 从读者视角判断是否有故事、有价值、有转化入口 | 审核口吻、重点、是否适合公开 | 长文内容确认可发布或可作为母稿 |
| 4. 视频演示包 | 面向 B 站展示真实使用过程 | 输出演示脚本、分镜、标题、简介、标签、封面文案，并准备上传素材包 | 判断是否能看懂、是否可信、是否有明确前后对比 | 决定是否录屏、是否出声、是否发布 | B 站发布包确认完成 |
| 5. 技术内容包 | 面向掘金建立技术可信度 | 输出掘金 Markdown 长文、技术亮点、实现取舍、结尾引导，并尽量进入草稿 | 判断是否技术可信、是否像真实开发者分享 | 审核技术表达和开源项目形象 | 掘金发布包确认完成 |
| 6. 社区轻量帖 | 面向 V2EX/即刻获取真实反馈 | 输出短帖、标题、评论区回复预案、风险提示，必要时辅助填入发布页 | 判断是否像广告、是否足够真诚、是否容易引发讨论 | 决定是否发布 | 社区内容确认，发布方式确认 |
| 7. 图文与短视频脚本 | 扩展到小红书、视频号 | 输出 6 张图文卡片文案、短视频脚本、分镜、封面标题，并准备可上传素材包 | 判断是否通俗、是否有视觉化表达、是否适合非技术用户 | 审核是否值得投入制作 | 图文/视频方向确认 |
| 8. 发布记录与复盘 | 判断效果和是否继续自动化 | 输出发布记录表、指标口径、复盘模板、下一轮建议，并尽量自动采集可得数据 | 判断数据是否能支持决策 | 补充真实发布数据和主观感受 | 完成复盘，决定是否沉淀 Skill |

每个阶段的固定交付格式：

1. 本阶段输出内容。
2. Codex 自评结论。
3. 需要你确认的问题。
4. 你反馈后的修改版。
5. 是否进入下一阶段的判断。

## 5. 验证指标与沉淀判断

第一轮重点看反馈质量和流程可复制性，不只看曝光量。

| 指标 | 观察目标 |
| --- | --- |
| GitHub Star | 是否有新增 Star |
| GitHub Issue | 是否有人提出真实问题、安装问题或功能建议 |
| GitHub Fork / Clone | 是否有人尝试安装、二次改造或本地试用 |
| 评论反馈 | 是否有人理解“不是摘要，是入库”的痛点 |
| 安装追问 | 是否有人询问配置 Vault、Cookie、字幕、ASR、本地视频等具体问题 |
| 内容修改次数 | 你每阶段需要反馈几轮 |
| 自动化价值 | 哪些环节明显节省时间 |
| 草稿成功率 | 哪些平台可以稳定自动进入草稿或待发布状态 |
| 发布风险 | 哪些平台不适合自动化发布 |

沉淀判断：

- 如果只是这次单个项目推广，不沉淀成 Skill。
- 如果多渠道内容包、审核清单、发布记录、平台草稿、复盘流程重复出现 3 次以上，建议创建 `marketing-publisher` Skill。
- 全局 AGENTS.md 只补充长期稳定的协作规则，不写具体渠道文案。
- 项目级 AGENTS.md 可记录当前项目的固定链接、目标用户、发布口径和禁用表达。
- 平台 API/MCP/CLI 的固定接入方式如果持续使用，应沉淀到独立 Skill 或项目脚本。
