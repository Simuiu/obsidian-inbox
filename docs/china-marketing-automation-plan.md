# Obsidian Inbox 中国市场外部引流与自动化推广协作计划

版本：v0.2 结构重构版  
日期：2026-05-08  
用途：用于确认推广方向、拆分执行阶段、生成首批内容物料，并逐步验证自动化推广流程。

## 1. 当前目标与原则

`Obsidian Inbox` 的基础产品形态已经具备：它可以把视频、文章、本地资料整理进 Obsidian Vault，生成结构化笔记、保留原始材料，并重建资料库索引。

当前阶段的重点不是把它包装成“又一个 AI 总结工具”，而是通过外部渠道找到第一批真实用户，验证这个定位是否成立：

> 给 Obsidian 装一个本地知识入库员：把视频、文章、本地资料变成可追溯、可索引、可复用的知识库。

第一阶段目标：

- 获得 100-300 个高质量早期用户关注。
- 为 GitHub 项目带来真实访问、Star、Issue 和试用反馈。
- 验证“AI 生成内容 + 人工审核 + 半自动发布 + 数据复盘”的推广流程。
- 明确哪些内容生产环节值得自动化，哪些发布动作必须保留人工确认。

核心原则：

- 先验证方向，再扩大执行。
- GitHub 是统一承接页，所有外部渠道都尽量导向 GitHub、Issue 或演示文档。
- 内容先由 Codex 生成和自评，再交给你最终确认。
- 不追求全平台自动发布，优先验证可控、低风险、能复盘的半自动流程。
- 不宣传绕过平台限制、登录、验证码、付费墙，也不宣传无人值守全自动运营。
- 如果同类推广流程重复 3 次以上，再考虑沉淀成 Skill。

## 2. 营销方向判断

### 2.1 核心判断

`Obsidian Inbox` 对外传播时，不能只讲“AI 可以总结视频和文章”，因为这个说法太常见，也容易让用户以为结果只是一次性摘要。

更清晰的传播重点是：

- 它不是一次性总结，而是内容入库工作流。
- 它不是云端知识库，而是 local-first 的 Obsidian 工作流。
- 它不是黑盒抓取工具，而是失败安全、可追溯、有资源归档的本地工具链。
- 它不是单个 prompt，而是 Codex Skill + Python 脚本 + Obsidian 索引结构。

### 2.2 目标用户

| 用户类型 | 典型痛点 | 推荐话术 |
| --- | --- | --- |
| Obsidian / Logseq / Notion 用户 | 收藏很多，但很少真正沉淀 | 不是给你更多摘要，而是帮你把资料放进以后还能找回来的结构里 |
| Codex / AI Agent 用户 | 想把 AI 能力沉淀成可重复工作流 | 这是一个可复用的 Codex Skill，不是一次性提示词 |
| 课程学习和专业研究用户 | 长视频、课程、会议资料难整理 | 一个本地视频进去，一篇带转写追溯的 Obsidian 深度笔记出来 |
| 开源工具爱好者 | 关心本地优先、可验证、可改造 | Vault、字幕、正文、原始材料默认留在本地 |
| 创作者 / 咨询 / 产品 / 投研人群 | 资料多、复盘难、后续查找难 | 把内容消费变成可复用的资料库资产 |

### 2.3 差异化表达

| 常见 AI 总结工具 | Obsidian Inbox |
| --- | --- |
| 生成一次性摘要 | 生成可长期维护的知识库条目 |
| 结果散落在对话里 | 写入 Obsidian Vault |
| 原文和转写容易丢 | `_resources/` 保留可追溯材料 |
| 只有一个摘要层级 | 普通总结 / 深度总结 |
| 很难后续查找 | 资料库索引 / 主题索引 / 用途索引 |
| 常把自动化讲成全自动 | 明确区分自动、半自动、人工确认边界 |

### 2.4 渠道优先级

| 渠道 | 作用 | 先后判断 | 自动化策略 |
| --- | --- | --- | --- |
| GitHub | 统一承接页，承接 Star、Issue、安装说明 | 第一优先级 | README、Topics、Release、Issue 模板可自动生成草稿 |
| B 站 | 建立可信度，展示完整演示 | 第一批内容 | 自动生成脚本、分镜、标题、简介，人工投稿 |
| 公众号 / 知乎 | 讲完整故事和方法论，沉淀长期内容 | 第一批内容 | 生成长文稿，人工审核发布 |
| 掘金 / CSDN / 开源中国 | 建立技术可信度 | 第二优先级 | 生成 Markdown 技术文，人工发布 |
| V2EX / 即刻 | 获取早期用户反馈 | 第二优先级 | 生成轻量发布帖和回复预案，人工发布 |
| 小红书 | 测试效率工具、知识管理人群兴趣 | 第三优先级 | 生成图文卡片文案和封面提示词 |
| 视频号 / 抖音 | 复用短视频素材，做前后对比 | 第三优先级 | 生成短视频脚本和字幕，人工发布或半自动上传 |

### 2.5 禁用表达

对外不使用这些说法：

- 全平台一键下载
- 自动绕过登录、验证码、付费墙
- 无人值守全自动运营
- 保证视频号 URL 直接下载
- 一键总结任何链接
- 替代人工判断

推荐统一表达：

> 这是一个 local-first 的 Codex + Obsidian 入库工作流。能自动化的自动化，涉及平台权限、发布审核、账号安全的部分保留人工确认。

## 3. 我们的角色分工

| 角色 | 职责 |
| --- | --- |
| Codex 执行者 | 生成渠道内容包、标题、正文、脚本、发布说明、发布检查清单、复盘模板 |
| Codex 评审者 | 从营销专家、目标用户、平台用户视角自评内容是否可信、是否像广告、是否能打动人 |
| 你 | 做最终方向确认、口吻确认、发布风险确认，以及是否进入下一阶段的决策 |
| 小龙虾 | 作为内容运营执行层，负责定期生成、改写、排期、提醒、复盘，不直接越权发布敏感平台内容 |

协作标准：

- Codex 交付给你的内容，必须已经过自评，不把粗糙初稿直接丢给你。
- 你反馈后，Codex 先修改当前阶段内容，不跳到下一阶段。
- 只有你确认“这一阶段可以”，才进入下一阶段。
- 小龙虾只做内容生产、改写、排期、提醒、复盘，不负责改核心代码、不绕过平台限制、不自动回复争议评论。
- 目标是让你每轮需要提的意见越来越少。

## 4. 全流程协作计划

| 阶段 | 阶段目标 | Codex 负责 | Codex 自评 | 你负责 | 进入下一阶段条件 |
| --- | --- | --- | --- | --- | --- |
| 1. 统一口径 | 定清楚所有渠道共用的定位、卖点和边界 | 输出统一定位、核心卖点、禁用表达、GitHub 引流口径 | 判断是否清楚、可信、不过度夸大 | 确认方向和表达边界 | 你确认口径可用于所有渠道 |
| 2. GitHub 承接页 | 先把所有渠道的落地点做好 | 优化 README 首屏、About、Topics、Release、Issue 模板、Star CTA | 判断新人是否能看懂、是否愿意 Star 或试用 | 确认 GitHub 形象和安装说明 | GitHub 页面可作为统一承接页 |
| 3. 长文母稿 | 用公众号/知乎长文讲完整故事 | 输出长文大纲、正文、标题候选、摘要、发布备注 | 判断是否有故事、有价值、有转化入口 | 审核口吻、重点和公开风险 | 长文可发布或可作为多平台母稿 |
| 4. B 站演示包 | 用视频降低理解成本 | 输出 5-8 分钟演示脚本、分镜、标题、简介、标签、封面文案 | 判断演示是否具体、节奏是否清楚、是否能建立信任 | 决定是否录屏、是否出声、是否发布 | B 站发布包确认完成 |
| 5. 技术内容包 | 面向开发者建立技术可信度 | 输出掘金/CSDN Markdown 长文、架构图说明、实现取舍、结尾引导 | 判断是否技术可信、是否像真实开发者分享 | 审核技术表达和开源项目形象 | 技术文发布包确认完成 |
| 6. 社区轻量帖 | 面向 V2EX/即刻获取真实反馈 | 输出短帖、标题、评论区回复预案、风险提示 | 判断是否像广告、是否真诚、是否容易引发讨论 | 决定是否人工发布 | 社区帖确认，发布方式确认 |
| 7. 图文与短视频 | 扩展到小红书、视频号、抖音 | 输出 6-9 页图文卡片文案、短视频脚本、字幕、封面提示词 | 判断是否通俗、是否视觉化、是否适合非技术用户 | 审核是否值得投入制作 | 图文/短视频方向确认 |
| 8. 发布记录与复盘 | 判断效果和是否继续自动化 | 输出发布记录表、指标口径、周复盘、下一轮选题建议 | 判断数据是否能支持决策 | 补充真实发布数据和主观感受 | 完成复盘，决定是否进入下一轮 |

每个阶段的固定交付格式：

1. 本阶段输出内容。
2. Codex 自评结论。
3. 需要你确认的问题。
4. 你反馈后的修改版。
5. 是否进入下一阶段的判断。

## 7. 自动化建设计划

### 7.1 自动化分级

| 级别 | 含义 | 当前阶段策略 |
| --- | --- | --- |
| L0 人工 | 最终发布确认、评论区高价值回复、平台敏感判断 | 必须保留 |
| L1 素材自动生成 | 标题、正文、标签、摘要、封面文案、脚本、SRT | 第一阶段必须完成 |
| L2 跨平台改写和排期 | 从一个母稿改写为多平台版本，生成发布时间建议和 checklist | 第一阶段重点验证 |
| L3 半自动发布 | 公众号进草稿箱、抖音申请权限后上传待发布、浏览器辅助填写 | 后续谨慎验证 |
| L4 自动复盘 | 汇总阅读、播放、点赞、收藏、评论、Star、Issue 等数据 | 第一阶段先用模板手动录入 |

### 7.3 小龙虾任务定位

建议把小龙虾定位为：

> 内容运营执行层：负责定期生成、改写、排期、提醒、复盘；不直接越权发布敏感平台内容。

每日任务：

- 检查 GitHub 是否有新 commit、issue、star 变化。
- 生成 1 条即刻/朋友圈/动态短内容候选。
- 检查评论和反馈是否需要人工回复。

## 8. 平台自动化边界

| 平台 | 自动生成内容 | 自动发布 | 建议 |
| --- | ---: | ---: | --- |
| GitHub | 是 | 是 | 可自动生成 release、issue 模板、README 更新稿，是否提交由你确认 |
| 公众号 | 是 | 草稿箱可自动 | 优先做“自动进草稿箱”，最终发布人工确认 |
| 抖音 | 是 | 有开放平台能力但需申请 | 权限通过后可半自动上传，发布前人工确认 |
| B 站 | 是 | 不建议默认自动 | 人工投稿更稳，封面、分区、标签要人工检查 |
| 小红书 | 是 | 不建议默认自动 | 不默认使用非官方 cookie 自动发帖 |
| 视频号 | 是 | 不建议默认自动 | 人工发布 |
| 知乎 | 是 | 不建议默认自动 | 人工发布，避免硬广 |
| 掘金 / CSDN | 是 | 可探索 | 先人工发布 |
| 即刻 / V2EX | 是 | 不建议自动 | 人工发布，重视互动 |

## 9. 验证指标与复盘

第一轮重点看反馈质量和流程可复制性，不只看曝光量。

| 指标 | 观察目标 |
| --- | --- |
| GitHub Star | 是否有新增 Star |
| GitHub Issue | 是否有人提出真实问题或功能建议 |
| GitHub Fork / Clone | 是否有人尝试安装或二次改造 |
| README 访问 | 外部渠道是否带来项目页访问 |
| 评论反馈 | 是否有人理解痛点、追问安装、提出替代方案 |
| 内容收藏 / 点赞 | 哪类内容更容易被目标用户保存 |
| 内容修改次数 | 你每阶段需要反馈几轮 |
| 自动化价值 | 哪些环节明显节省时间 |
| 发布风险 | 哪些平台不适合自动化发布 |

复盘输出：

- 本周发布了什么。
- 哪些渠道有反馈。
- 哪些内容被理解，哪些内容没有被理解。
- 是否有人复述出“不是摘要，是入库”的定位。
- 哪些物料下周复用。
- 哪些自动化值得继续做。
- 哪些规则需要写进项目 AGENTS.md 或沉淀成 Skill。

## 10. 需要确认的问题

1. 首发内容是否以 GitHub 为唯一承接页？
2. 首发案例用哪一种资料？
   - B 站技术视频
   - 本地课程视频
   - 微信公众号文章
   - 不建议首发用交易类视频，容易偏离工具定位
3. 首发视频是否愿意出声？
   - 不露脸不出声：录屏 + 字幕 + AI 配音
   - 不露脸出声：录屏 + 口播
   - 露脸：产品可信度更高，但制作成本更高
4. 是否建立反馈群？
   - 不建群：只导 GitHub Issue
   - 建群：公众号/小红书更容易转化，但管理成本增加
5. 自动发布边界你能接受到哪一级？
   - A：只自动生成物料，全部人工发布
   - B：公众号进草稿箱，其他人工
   - C：部分平台浏览器辅助填写，人工点发布
   - D：有 API 的平台自动发布，无 API 的平台人工发布

## 11. 沉淀判断

- 如果只是这次单个项目推广，不沉淀成 Skill。
- 如果多渠道内容包、审核清单、发布记录、复盘流程重复出现 3 次以上，建议创建 `marketing-publisher` Skill。
- 全局 AGENTS.md 只补充长期稳定的协作规则，不写具体渠道文案。
- 项目级 AGENTS.md 可记录当前项目的固定链接、目标用户、发布口径和禁用表达。
- 小龙虾相关固定流程如果持续使用，应沉淀到独立 Skill，而不是塞进普通计划文档。
