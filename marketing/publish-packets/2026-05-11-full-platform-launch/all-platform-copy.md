## 2. 微信公众号

### 标题

AI 总结之后，知识应该去哪？我做了一个 Obsidian 本地入库工具

### 摘要

Obsidian Inbox 不是又一个 AI 总结工具，而是一套 local-first 的 Codex + Obsidian 入库工作流：把视频、文章、本地资料整理成可追溯、可索引、可复用的知识库。

### 正文

很多人现在已经习惯用 AI 总结文章、视频和课程。

但我遇到的真实问题是：总结完之后，它去哪了？

它可能停留在一次聊天记录里，可能散落在某个收藏夹里，也可能被复制到一个临时文档里。过几天再想找，还是要重新翻链接、重新问 AI、重新整理。

所以我做了一个小工具：`Obsidian Inbox`。

它不是为了再生成一段更漂亮的摘要，而是想解决“摘要之后”的问题：

- 这条资料应该放在哪里？
- 半年后还能不能找回来？
- 原文、字幕、转写能不能追溯？
- 快速收藏和深度学习能不能分开？
- 如果平台抓不到正文或字幕，工具会不会编造一篇看似完整的笔记？

`Obsidian Inbox` 的定位很简单：

> 把 Codex 变成你的本地知识入库员：链接进来，Obsidian 笔记、原始材料和可点击索引自动生成。

它的基本流程是：

```text
链接或本地文件
-> 本地脚本采集正文/字幕/转写
-> 生成任务包
-> Codex 读取 compact.md
-> 写入结构化 Obsidian 笔记
-> 原始材料进入 _resources
-> 重建资料库索引、主题索引、用途索引
```

我刻意把它做成 local-first。

也就是说，Vault、字幕、正文、转写、本地视频路径、原始材料，默认都在你自己的机器上处理。它不会默认把你的 Obsidian Vault 上传到云端，也不会尝试绕过登录、验证码、付费墙或私有内容限制。

如果抓不到正文、字幕或可靠转写，它不会硬编一篇总结，而是保留失败状态，等你补充材料。

目前它支持几类场景：

1. B 站视频：读取元数据和字幕，生成普通或深度 Obsidian 笔记。
2. 微信公众号公开文章：抓取公开正文，转成 Markdown，并保留原始材料。
3. 本地视频/音频：导入本地文件，结合转写或 ASR 结果生成笔记。
4. 普通网页和 YouTube：保守支持，取决于正文和字幕是否可访问。
5. 微信视频号、小红书、抖音等：优先走本地导出或本地捕获材料，再进入入库流程。

我最在意的不是“能不能一次性总结”，而是这条资料能不能稳定变成知识库里的一个资产。

所以每条资料会有一个主笔记，也会有 `_resources` 目录保存相关材料。索引层面会维护三个入口：

- 资料库索引：所有入库资源的总入口。
- 主题索引：按主题查找。
- 用途索引：按学习、实践、素材等用途查找。

笔记也分成两种模式：

- 普通总结：适合快速入库和第一遍理解。
- 深度总结：适合课程、教学、系统学习、本地视频和需要长期复用的内容。

举个例子，如果你把一个本地课程视频交给它，理想结果不是一段“本视频主要讲了什么”，而是一篇带问题链、概念地图、机制拆解、实践路径和来源追溯的 Obsidian 深度笔记。

我希望它服务的人群也很明确：

- 已经使用 Obsidian，但收藏和资料整理压力很大的人。
- 经常用 AI 总结内容，但结果散落在聊天窗口里的人。
- 看课程、会议、长视频，需要沉淀复盘材料的人。
- 想把 Codex 从一次性问答变成可重复工作流的人。
- 关心本地优先、Markdown、可追溯材料的知识管理用户。

它现在还处在早期阶段，不是一个“全平台一键导入”的万能工具。

我更愿意把它说清楚：

> 能自动化的自动化；涉及平台权限、账号安全、内容判断和发布审核的部分，保留人工确认。

如果你也有类似需求，可以先看 GitHub 项目：

https://github.com/Simuiu/obsidian-inbox

如果这个方向对你有用，欢迎 Star。也欢迎在 Issue 里告诉我：

- 你希望 Obsidian 笔记长什么样？

对我来说，这个项目真正想解决的问题不是“AI 能不能总结”，而是：

我的答案是：回到你自己的知识库里，带着来源，带着索引，带着以后还能复用的结构。

### 摘要

这篇回答适合发在知乎问题下，重点讲“AI 总结之后的入库工作流”，弱化硬广，最后自然引导 GitHub。

### 正文

我现在越来越觉得，AI 总结本身不是知识管理里最难的一步。

真正难的是：总结之后，这些内容能不能回到一个长期可维护、可检索、可复用的系统里。

很多人的流程是这样的：

1. 看到一篇文章或一个视频。
2. 扔给 AI 总结。
3. 得到一段看起来不错的摘要。
4. 摘要留在聊天记录里。
5. 几天后忘了它在哪里。

这就会导致一个问题：AI 提高了“当下理解”的效率，但没有解决“长期沉淀”的问题。

所以我会更建议把 AI 总结看成知识管理链路中的一个环节，而不是终点。

一个更完整的流程应该是：

```text
资料进入
-> 提取正文/字幕/转写
-> AI 生成结构化笔记
-> 保留原始材料
-> 写入知识库
-> 建立索引
-> 后续可以复盘、引用、二次加工
```

我自己为这个思路做了一个开源小工具：`Obsidian Inbox`。

它的目标不是再做一个“总结得更好”的工具，而是把 Codex 接进 Obsidian，让视频、文章和本地资料可以变成结构化 Markdown 笔记。

它有几个设计原则：

- local-first：Vault、转写、正文和原始材料默认留在本地。
- Obsidian-native：输出 Markdown、frontmatter、Wiki links 和索引。
- 可追溯：原始材料进入 `_resources`，笔记不只是孤立摘要。
- 失败安全：抓不到正文、字幕或可靠转写时，不编造总结。
- 分层总结：普通总结用于快速入库，深度总结用于课程和系统学习。

我觉得这类工具真正适合的人，不是偶尔让 AI 总结一篇文章的人，而是这些人：

- 长期使用 Obsidian 做资料库的人。
- 收藏大量视频、文章、课程，但经常没有复盘的人。
- 希望 AI 输出能沉淀进自己知识系统的人。
- 关心本地优先和可验证工作流的人。

当然，它不是万能的，也不会绕过平台限制。涉及登录、验证码、付费墙、私有内容的场景，仍然应该遵守平台规则，优先使用自己可访问或本地导出的材料。

如果你也在用 Obsidian，并且遇到“收藏很多但沉淀很少”的问题，可以看看这个项目：

https://github.com/Simuiu/obsidian-inbox

我更推荐把它理解成一个思路：

> 不要只追求让 AI 总结得更快，而是让 AI 生成的东西有地方沉淀。

## 4. 掘金

### 标题

如何设计一个 local-first 的 Codex + Obsidian 内容入库工作流

### 摘要

本文面向开发者，解释 Obsidian Inbox 为什么不是普通摘要工具，而是由采集脚本、任务包、Codex Skill、资源归档和索引重建组成的本地内容入库工作流。

### 正文

这篇文章介绍一个我最近在做的开源项目：`Obsidian Inbox`。

项目地址：

https://github.com/Simuiu/obsidian-inbox

它解决的问题不是“如何让 AI 总结一篇文章”，而是另一个更工程化的问题：

> 如何把外部内容稳定地变成 Obsidian 里可追溯、可索引、可复用的 Markdown 笔记？

## 设计目标

很多 AI 总结工具的结果停留在对话里。对个人知识管理来说，这个结果很脆弱：

- 缺少固定存放位置。
- 缺少来源追溯。
- 缺少统一 frontmatter。
- 缺少索引。
- 缺少失败状态。
- 很难批量复盘和二次加工。

所以 `Obsidian Inbox` 采用了一个更偏工作流的设计：

```text
source
-> capture script
-> task package
-> compact.md
-> Codex Skill
-> Obsidian note
-> _resources archive
-> rebuilt indexes
```

## 核心结构

任务包通常包含：

- `meta.json`：平台、标题、作者、来源 URL、状态等元数据。
- `compact.md`：给 Codex 读取的低 token 摘要材料。
- `prompt.md`：当前任务的写作要求。
- 原始正文、字幕或转写材料。

Codex 默认只读取 `meta.json`、`compact.md`、`prompt.md`，避免把完整字幕或正文塞进上下文。

写入 Obsidian 后，每条资源会形成类似结构：

```text
00_资料库/YYMMDD 短标题/YYMMDD 短标题.md
00_资料库/YYMMDD 短标题/_resources
```

索引则维护：

```text
00_资料库/000 资料库索引/000 资料库索引.md
00_资料库/001 主题索引/001 主题索引.md
00_资料库/002 用途索引/002 用途索引.md
```

## 为什么要保留 `_resources`

AI 生成内容最大的问题之一是“看起来对，但无法回到来源”。

所以这个项目会把原始材料归档到 `_resources`，主笔记只保留总结、重点、时间点、行动清单和资源链接。

这样做有几个好处：

1. 主笔记保持可读。
2. 原始材料仍然可追溯。
3. 后续可以重新总结或做深度笔记。
4. 不需要把完整字幕堆进主笔记。

## 普通总结和深度总结

普通总结适合快速入库：

- 一句话总结。
- 重点速览。
- 核心内容。
- 时间点或段落线索。
- 行动清单。

深度总结适合课程、教学和系统学习：

- 核心结论。
- 问题链。
- 概念地图。
- 机制拆解。
- 对比表。
- 常见误区。
- 实践路径。
- 二次加工卡片。

这两个模式会写入 frontmatter，例如：

```yaml
summary_mode: "深度"
```

索引页也能显示总结模式，便于后续筛选。

## 失败安全

我不希望这个工具在没有可靠材料时编造总结。

因此如果正文、字幕或转写抓取失败，任务会保留失败状态。比如微信视频号、本地视频等场景，如果没有转写或 ASR 结果，就不会生成假总结。

这也是它和一些“任何链接都能总结”的工具不同的地方。

## 当前支持

- Bilibili 视频字幕入库。
- 微信公众号公开文章正文入库。
- 本地视频/音频导入。
- 普通网页保守抽取。
- YouTube 字幕路径部分支持。
- 微信视频号、小红书、抖音等更适合走本地导出材料。

## 适合谁

这个项目适合：

- Obsidian 重度用户。
- 希望把 Codex 变成本地工作流的人。
- 有大量课程、文章、会议资料要沉淀的人。
- 关心 local-first、Markdown 和可追溯材料的人。

项目地址：

https://github.com/Simuiu/obsidian-inbox

## 10. GitHub Release

### 标题

Obsidian Inbox v0.1：把视频、文章和本地资料整理进 Obsidian

### 正文

`Obsidian Inbox` 是一个 local-first 的 Codex Skill，用来把外部内容整理成结构化 Obsidian 笔记。

它不是一次性 AI 摘要工具，而是一套内容入库工作流：本地脚本负责采集、归档和索引，Codex 负责理解内容并生成普通总结或深度总结。

当前能力：

- Bilibili 视频字幕入库。
- 微信公众号公开文章正文入库。
- 本地视频/音频导入，支持转写或 ASR 路径。
- 普通总结和深度总结两种笔记模式。
- 原始材料归档到 `_resources/`。
- 自动重建资料库索引、主题索引和用途索引。
- 抓不到字幕、正文或可靠转写时保留失败状态，不编造总结。

项目地址：

https://github.com/Simuiu/obsidian-inbox

## 11. 发布顺序建议

1. 先发 V2EX 或即刻，收第一波真实反馈。
2. 再发公众号/知乎长文，形成可长期引用内容。
3. 然后发掘金技术文，建立开发者可信度。
4. B 站、小红书、视频号等视频/图文内容，可以在录屏素材准备好后发布。

## 12. Codex 自评结论

这组物料已经覆盖第一轮计划中的主要平台。整体口径保持一致：不强调万能抓取，不夸大全自动，而是强调 local-first、Obsidian-native、可追溯和内容入库。

需要你后续重点检查：

1. 公众号长文是否太技术，是否要更口语。
2. 知乎是否需要减少项目名露出，进一步降低广告感。
3. B 站是否需要补真实录屏分镜。
4. 小红书是否需要进一步做成图片资产，而不是只给卡片文案。
