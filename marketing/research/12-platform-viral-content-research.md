# 12 Obsidian Inbox 平台爆款内容调研与改写策略

## 1. 这次为什么要重写

上一版发布助手能复制标题、正文和图片路径，但对真实发布不够友好：

- 配图只是素材列表，没有告诉你该放在正文哪里。
- 长文内容偏短，层次不够，读者看不到“问题、解决、演示、边界、行动”的完整链路。
- 标题偏长，像项目介绍，不像平台用户会点开的内容。
- 各平台文案差异不够明显，容易把同一篇稿子机械搬运到所有平台。

这次改写的目标不是“标题党”，而是让每个平台都像原生内容：

- 微信公众号：完整故事 + 分段小标题 + 图文节奏 + 阅读原文引导。
- 知乎：先回答问题，再给个人实践和工具链接，弱化硬广。
- 掘金：技术可信，讲设计取舍、目录结构、失败边界。
- V2EX：真诚自荐，短、具体、可讨论，不伪装成经验贴。
- 即刻：像动态，不像公告，重点引发同温层讨论。
- 小红书：首图强钩子，后续卡片按“痛点-流程-结果-边界-行动”展开。
- B 站：标题和封面先讲收益，简介补充项目链接和章节。
- 视频号：前 3 秒钩子和封面优先，文案短平快。

## 2. 平台内容特征

| 平台 | 内容特征 | 这次怎么借鉴 |
| --- | --- | --- |
| 微信公众号 | 标题和摘要决定打开，正文需要结构化、配图打断长段落；运营资料普遍建议标题控制在较短范围、摘要提炼精华、配图强化表达。 | 标题缩短到 20 字上下；正文改成 7 个明确小节；每 300-500 字安排一张图位。 |
| 知乎 | 高赞回答常见“结论先行、证据/场景支撑、给可复刻方法”的结构，段落要短。 | 开头直接回答“AI 总结有没有用”；中段给工作流；最后自然放 GitHub。 |
| 掘金 | 技术读者关注架构、边界、代码结构、失败处理、是否开源可用。 | 改成技术设计文：问题背景、架构、数据结构、关键取舍、适用/不适用。 |
| V2EX | 社区对推广敏感，开源自荐要直接说明“做了什么、为什么做、想听什么反馈”。 | 用短帖，不做夸张卖点；主动说明早期、限制和求反馈点。 |
| 即刻 | 信息流动态更适合个人视角、即时感、一个清晰问题，引导评论区经验交流。 | 用“我最近把 AI 总结往前推进了一步”的口吻，配 5 图说明。 |
| 小红书 | 首图和标题非常关键，爆款拆解普遍强调封面、标题、正文结构协同；图文卡片要降低理解门槛。 | 6 张卡片按顺序上传，每张图都有一句发布说明和正文对应段落。 |
| B 站 | 知识/科技区封面要少字大字，标题要问题词 + 方案词，开头 Hook 要尽快讲清收益。 | 使用最终封面图，视频简介加章节和 GitHub；置顶评论引导 Star/Issue。 |
| 视频号 | 封面和前 3 秒对停留很重要，文案前几行要直接给结果；很多运营建议短标题、强动作词、第一帧做标题图。 | 视频开头就是“AI 总结之后去哪”；文案控制短段，封面/卡片按上传顺序给出。 |

## 3. 改写原则

1. 标题从“项目介绍型”改成“问题/结果型”。
2. 正文从“这是一个工具”改成“我遇到的问题 -> 为什么旧方法不够 -> 我怎么设计 -> 你能怎么用”。
3. 每个平台都保留限制说明：不绕过登录、验证码、付费墙，不承诺全平台一键。
4. 每个平台都放 GitHub 承接链接，但位置不同：
   - 长文：文末 + 阅读原文。
   - 技术文：开头和结尾各一次。
   - 社区帖：正文中段或末尾一次。
   - 视频：简介 + 置顶评论。
5. 每篇/每条内容不少于 5 张配图，HTML 中明确图位。
