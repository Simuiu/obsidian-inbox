# Obsidian Inbox 统一推广口径

## 1. 本阶段目标

本阶段先统一所有平台共用的表达口径，避免后续每个平台单独写时方向发散。

统一目标：

- 让用户快速理解 `Obsidian Inbox` 是什么。
- 明确它不是普通 AI 总结工具，而是 Obsidian 内容入库工作流。
- 所有外部内容统一导向 GitHub 项目页、Issue 或后续演示文档。
- 保持 local-first、可追溯、失败安全的可信表达。

## 2. 一句话定位

主推中文版本：

> 把 Codex 变成你的本地知识入库员：链接进来，Obsidian 笔记、原始材料和可点击索引自动生成。

更短版本：

> 给 Obsidian 装一个本地知识入库员。

面向开发者版本：

> 一个 local-first 的 Codex Skill，把视频、文章和本地资料整理成可追溯的 Obsidian Markdown 笔记。

英文版本：

> Turn Codex into a local knowledge librarian for Obsidian.

## 3. 核心卖点

| 卖点 | 对用户的意义 | 推荐表达 |
| --- | --- | --- |
| Local-first | Vault、转写、正文、原始材料默认留在本地 | 不把你的知识库上传到云端黑盒里 |
| Obsidian-native | 输出 Markdown、frontmatter、Wiki links 和索引 | 不是一段聊天摘要，而是一条 Obsidian 资料库记录 |
| Source-traceable | `_resources/` 保留原始材料 | 以后复盘时能回到来源，不怕摘要断章取义 |
| Failure-safe | 抓不到正文或字幕时不编造 | 不生成看似完整但来源不可靠的假笔记 |
| Standard / Deep | 普通总结和深度总结分层 | 快速收藏和系统学习可以走不同模式 |
| Codex-powered | AI 负责理解和写作，脚本负责采集、归档和索引 | 把一次性 prompt 变成可重复工作流 |

## 4. 目标用户

| 用户 | 典型痛点 | 主要话术 |
| --- | --- | --- |
| Obsidian 用户 | 收藏很多，但资料很难长期复用 | 不是再给你一个模板，而是补上“内容入库”这一步 |
| AI 工具用户 | 用 AI 总结很多，但结果散在对话里 | 把 AI 输出直接沉淀进 Obsidian，而不是停在聊天窗口 |
| 课程学习者 | 长视频、课程、会议内容难整理 | 一个视频进去，一篇带来源追溯的深度笔记出来 |
| 开源工具用户 | 关心本地优先、可验证、能改造 | 结构透明，脚本和输出都能本地检查 |
| 创作者/研究者 | 资料多，后续查找和复盘成本高 | 把内容消费变成可复用的资料库资产 |

## 5. 禁用表达

不要使用这些说法：

- 全平台一键下载。
- 自动绕过登录、验证码、付费墙。
- 无人值守全自动运营。
- 一键总结任何链接。
- 保证视频号 URL 直接下载。
- 替代人工判断。

替代表达：

> 能自动化的自动化；涉及平台权限、账号安全、发布审核和内容判断的部分，保留人工确认。

## 6. GitHub 引流口径

短 CTA：

> 如果你也想把视频、文章和本地资料稳定沉淀进 Obsidian，欢迎 Star 这个项目，并在 Issue 里告诉我你最想支持哪类资料。

长 CTA：

> Obsidian Inbox 还在早期阶段。如果你正在用 Obsidian 做知识管理，或者想把 Codex 变成可重复执行的本地工作流，可以先 Star 项目，再用 Issue 反馈你最常需要入库的内容类型。

Issue 引导：

- 支持新平台：提交平台 URL、是否需要登录、希望抓取哪些内容。
- 抓取失败：提交命令、平台、失败状态、是否涉及登录/验证码/付费内容。
- 笔记结构建议：说明你希望 Obsidian 笔记长什么样。

## 7. Codex 自评结论

这版口径可以作为所有渠道的母口径使用。它把重点放在“入库工作流”而不是“AI 总结”，也保留了平台边界和隐私边界。

需要你确认的问题：

1. “本地知识入库员”这个说法是否作为主口号固定使用？
2. GitHub CTA 是否直接引导 Star + Issue？
3. 是否要在中文渠道里弱化 “Codex Skill” 这个术语，改成更小白的“Codex 工作流”？
