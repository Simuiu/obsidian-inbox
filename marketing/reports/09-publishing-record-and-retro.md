## 2. 评论反馈归类

| 类型 | 典型反馈 | 处理方式 |
| --- | --- | --- |
| 安装问题 | 依赖、配置、Vault 路径、Codex skill 使用 | 补 README / FAQ |
| 平台支持 | 想支持某个新平台 | 引导提交 Issue |
| 隐私疑问 | 是否上传 Vault、cookies、视频 | 强调 local-first 和安全边界 |
| 笔记结构 | 想要更短/更深/更多字段 | 进入 note template backlog |
| 自动化质疑 | 是否绕过平台限制 | 明确不绕过，失败安全 |

## 3. 每周复盘问题

1. 哪个平台带来最多有效反馈？
2. 哪个平台只是曝光，没有真实试用意向？
3. 用户是否理解“不是摘要，是入库”？
4. 哪些图文素材表现更好？
5. 是否有人提出具体安装或使用问题？
6. GitHub Star、Issue、Fork 是否有变化？
7. 下一周应该继续哪个平台，暂停哪个平台？

## 4. 下轮调整建议

| 观察 | 调整动作 |
| --- | --- |
| 用户看不懂 Codex Skill | 改成“Codex 工作流”或“本地入库员” |
| 用户担心隐私 | 单独做 local-first FAQ |
| 技术用户关注实现 | 加一篇架构拆解 |
| 小红书收藏高但点击少 | 强化封面和评论区 GitHub 引导 |
| B 站完播低 | 缩短演示，先展示结果再讲流程 |

## 5. Codex 自评结论

这份模板可以直接用于第一轮发布后的数据记录。第一轮不要只看阅读量，更要看是否有人追问安装、提交 Issue 或复述出“内容入库”的定位。
