# Obsidian Inbox 发布能力确认表

## 1. 本阶段目标

先确认各平台能做到哪一级发布自动化，再决定后续内容包如何制作。

自动化等级：

| 等级 | 含义 |
| --- | --- |
| 自动 | Codex 可通过 API、MCP、CLI 或可靠工具直接创建草稿、发布包、Release、Issue 等 |
| 半自动 | Codex 生成完整内容和检查清单，你复制/确认/点击发布 |
| 人工 | 平台风险较高或缺少稳定接口，Codex 只准备材料 |

安全原则：

- 不在对话里展示 token、cookie、session、auth 文件。
- 需要授权时，先说明权限、用途、保存位置和撤销方式。
- 默认优先创建草稿、待发布内容或本地发布包，不直接最终发布。

## 4. 第一轮执行顺序

建议先执行低风险、低依赖的平台：

1. GitHub 承接页：README、Topics、Issue 模板、Release 草稿。
2. 公众号/知乎长文母稿：先写内容，不急着接接口。
3. 掘金技术文：先写 Markdown 发布包。
4. V2EX/即刻短帖：先写短帖和回复预案。
5. B 站演示脚本：先准备脚本和素材清单。

等第一轮内容确认后，再逐个平台验证草稿自动化。
